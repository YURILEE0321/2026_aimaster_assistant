# Frontend 개발 규칙 — Knowledge Space 생성

본 프로젝트는 Design Component(DC) 아키텍처로 구현된 단일 화면 애플리케이션 프로토타입이다(`Knowledge Space 생성.dc.html`). 이후 실제 프레임워크(React 등)로 이관하더라도 아래 원칙을 유지한다.

## 1. 스타일링 원칙
- **인라인 스타일 우선.** 클래스 기반 스타일시트나 전역 CSS를 만들지 않는다. 컴포넌트/화면 단위로 스타일이 즉시 보여야 스트리밍/점진적 렌더링이 가능하다.
- 색상/타이포는 `design-spec.md`의 토큰(`--bg`, `--text`, `--accent-text`, ink-rgb 투명도 패턴)만 사용한다. 임의로 새 색을 추가하지 않는다.
- 다크/라이트 테마는 CSS 커스텀 프로퍼티(`--bg`, `--text`, `--accent-text`, `--ink-rgb`)를 런타임에 갱신하는 방식을 유지한다. 컴포넌트 내부에서 다크/라이트를 각각 하드코딩하지 않는다.
- 모달·드롭다운 표면은 테마와 무관하게 라이트 고정이라는 기존 의도를 유지한다(색상 값은 `design-spec.md` §0 참조). 이유: 오버레이는 항상 고대비 카드로 노출.
- 폰트는 `design-spec.md` §0에 정의된 3종(제목/본문/라벨용)만 사용한다. 새 폰트 추가 금지.

## 2. 상태 관리
- 화면은 5개의 주요 스테이지로 나뉜다: 빈 상태(empty) / 목록·상세(list) / 검토(review), 그리고 오버레이(모달, 유저메뉴). 스테이지 전환은 파생 상태(`isEmptyStage`, `isListStage`, `isReviewStage`)로 계산하고, 원본 상태는 최소 소스(예: `spaces`, `activeSpaceId`, `activeReview`)로만 보관한다.
- Space별 데이터(파일, 문서, 채팅)는 반드시 해당 `space.id`에 스코프해서 갱신한다(`updateActiveFiles`, `updateActiveChat` 패턴 유지) — Space 간 데이터가 섞이지 않도록 한다.
- 비동기 타이머(분석 진행 시뮬레이션)는 `this.timers[id]`에 보관하고, 파일 삭제/초기화 시 반드시 `clearTimeout`으로 정리한다. 메모리 누수·유령 상태 업데이트를 방지한다.
- 파생값(배지 색상, 라벨, 버튼 표시 여부 등)은 `renderVals()`에서 계산해 템플릿에는 이미 계산된 문자열/불리언만 넘긴다. 템플릿 홀에 조건식·연산을 넣지 않는다.

## 3. 템플릿 작성 규칙
- 반복은 `<sc-for>`, 조건 분기는 `<sc-if>`만 사용한다. JS 삼항연산자나 `.map()`을 템플릿에 직접 쓰지 않는다.
- 리스트 표시용 값은 항상 사전 가공된 배열(id, label, 색상, 콜백 포함한 객체)로 넘긴다. 예: `spaceList`, `uploadedFiles`, `wikiDocs`.
- 모든 `<sc-for>`/`<sc-if>`에 `hint-placeholder-count` / `hint-placeholder-val`을 지정해 스트리밍 중 레이아웃 시프트를 방지한다.
- 이벤트 핸들러는 `renderVals()`에서 바인딩된 함수로 노출하고, 클로저 내부에서 필요한 id만 캡처한다(예: `onSelect: () => this.selectSpace(sp.id)`).

## 4. 네이밍/데이터 규칙
- 상태값 문자열은 영문 스네이크/카멜(`pending`, `approved`, `rejected`, `analysis_failed`)로 유지하고, 사용자에게 보이는 한글 라벨은 `renderVals()`의 매핑 함수(`statusStyle` 등)에서만 생성한다 — 상태 판별 로직과 표시 문구를 분리한다.
- `space_id`, `file_id`, `document_id` 등 식별자는 API 명세(`api-spec.md`)와 동일한 접두사 규칙(`spc_`, `file_`, `doc_`)을 따른다.
- 사용자 대상 카피는 모두 한국어, 정중체(해요체) 유지. 에러/경고 메시지도 동일 톤 유지(예: "~에 오류가 발생했어요").

## 5. 접근성 & 상호작용
- 클릭 가능한 요소는 `<button>` 또는 명확한 `onClick`을 가진 요소로 만들고, `style-hover`로 호버 피드백을 항상 준다.
- 배경 클릭으로 닫히는 오버레이(모달, 유저메뉴)는 반드시 `stopProp`으로 내부 클릭 전파를 막아 의도치 않게 닫히지 않게 한다.
- 입력 검증 에러(Space 이름 필수 등)는 인라인 메시지로 즉시 보여주고, 입력이 재개되면(`onInput`) 에러를 즉시 해제한다.
- 파괴적 동작(반려 확정, 전체 초기화)은 사유 입력 또는 별도 확인 없이 즉시 실행하지 않도록 확인 단계(반려 사유 입력 폼 등)를 유지한다.

## 6. 신규 화면/기능 추가 시 체크리스트
1. 새 색상이 필요하면 기존 팔레트에서 파생(oklch/투명도 조정)했는지 확인 — 새 원색 추가 금지.
2. 새 상태는 `isXxxStage`/`isXxxView` 패턴으로 파생 플래그를 만들고, 배타적 조건들의 합이 항상 참이 되는지 확인(빈 화면 방지).
3. 리스트/폼에 로딩·빈 상태·에러 상태 3종을 모두 설계했는지 확인(예: 업로드 없음/분석중/실패).
4. Space 간 데이터 격리, 타이머 정리, 재렌더 시 placeholder(hint-*) 누락 여부를 확인한다.
5. 카피 톤(해요체)과 아이콘 사용(이모지는 기존에 쓰인 것과 동일한 절제된 수준으로만) 일관성을 확인한다.
