Wiki Builder 호출 비동기 구조 개선 요청

현재 backend-proxy에서 파일 업로드 후 Wiki Builder의 ingest API를 호출하고, Wiki 생성이 완료될 때까지 HTTP 응답을 기다리는 동기식 구조로 구현되어 있다.

문서 분석 및 LLM 호출 시간이 길어질 경우 다음과 같은 httpx.ReadTimeout이 발생하고 있다.

httpcore.ReadTimeout
httpx.ReadTimeout
builder ingest call failed

현재 Frontend는 다음 API를 polling하여 파일 처리 상태를 확인하고 있다.

GET /api/v1/files/{file_id}

Wiki 생성은 수십 초에서 수분 이상 걸릴 수 있으므로, Proxy Backend가 Builder의 작업 완료까지 하나의 HTTP 요청을 계속 유지하지 않도록 비동기 작업 구조로 변경한다.

⸻

1. 목표 구조

다음 흐름으로 변경한다.

Frontend
   │
   │ 파일 업로드
   ▼
Proxy Backend
   │
   ├─ 파일 저장
   ├─ 상태 저장: PROCESSING
   │
   └─ Builder 작업 요청
            │
            ▼
       Builder API
       작업 등록
            │
            ├─ 즉시 202 Accepted 반환
            │    job_id
            │
            ▼
       비동기 Wiki 생성
            │
            ├─ Extract
            ├─ Normalize
            ├─ Translate (필요 시)
            ├─ Enrich
            ├─ Verify
            └─ Markdown 생성
            │
            ▼
       작업 완료
            │
            ▼
       Proxy Backend로 결과 전달
            │
            ▼
       Wiki 및 상태 저장

Frontend는 기존과 같이 다음 API를 polling하여 상태를 확인한다.

GET /api/v1/files/{file_id}

⸻

2. 현재 문제

현재 구조는 다음과 같다.

Proxy Backend
   │
   │ POST Builder /ingest
   ▼
Wiki Builder
   │
   │ 수십 초~수분 동안 문서 분석 및 LLM 실행
   ▼
결과 반환

Proxy Backend의 HTTP Client가 Builder의 최종 결과를 기다리는 동안 ReadTimeout이 발생한다.

단순히 timeout 값을 늘리는 방식보다는 Builder의 ingest 요청을 비동기 Job 방식으로 변경한다.

⸻

3. Builder ingest API 변경

기존 ingest API가 Wiki 생성 완료 후 결과를 반환하고 있다면, 다음과 같이 변경한다.

요청

POST /api/v1/ingest

예시:

{
  "file_id": "file_ce592ab151",
  "source_url": "http://backend-proxy/api/v1/files/file_ce592ab151/content",
  "callback_url": "http://backend-proxy/api/v1/builder/callback"
}

실제 기존 DTO와 필드가 다르면 현재 프로젝트 구조를 최대한 유지하고 필요한 필드만 추가한다.

응답

Wiki 생성을 완료할 때까지 기다리지 않고 즉시 반환한다.

HTTP/1.1 202 Accepted
{
  "job_id": "job_123456",
  "file_id": "file_ce592ab151",
  "status": "QUEUED"
}

⸻

4. Builder 내부 비동기 처리

Builder는 ingest 요청을 받으면 다음 작업을 수행한다.

1. job_id 생성
2. 작업 상태를 QUEUED로 등록
3. Wiki 생성 작업을 비동기로 실행
4. API에는 즉시 202 Accepted 반환

PoC 단계에서는 현재 프로젝트 구조를 고려하여 가장 단순한 구현을 사용한다.

우선순위:

1순위: FastAPI BackgroundTasks 또는 asyncio 기반 비동기 Task
2순위: 현재 프로젝트에 이미 존재하는 Job Queue 사용
3순위: Redis/Celery 신규 도입

현재 프로젝트에 Redis/Celery가 없다면 이번 변경을 위해 새 인프라를 추가하지 않는다.

단, 프로세스 재시작 시 Background Task가 유실될 수 있다는 점은 코드 주석 또는 문서에 남긴다.

⸻

5. Job 상태 관리

Builder 작업 상태를 관리한다.

최소 상태:

QUEUED
PROCESSING
COMPLETED
FAILED

가능하다면 처리 단계를 별도로 관리한다.

EXTRACTING
NORMALIZING
TRANSLATING
ENRICHING
VERIFYING
GENERATING

예:

{
  "job_id": "job_123456",
  "file_id": "file_ce592ab151",
  "status": "PROCESSING",
  "stage": "ENRICHING"
}

상태값은 기존 프로젝트의 Enum 또는 상태 관리 방식을 우선 재사용한다.

⸻

6. Proxy Backend 변경

Proxy Backend는 Builder ingest API의 최종 Wiki 결과를 기다리지 않는다.

Builder 호출:

POST Builder /ingest

응답:

{
  "job_id": "job_123456",
  "status": "QUEUED"
}

Proxy Backend는 다음 정보만 저장한다.

file_id
builder_job_id
processing_status

예:

file_id             = file_ce592ab151
builder_job_id      = job_123456
status              = PROCESSING

Builder 호출이 정상적으로 작업 등록되면 장시간 HTTP 연결을 유지하지 않는다.

⸻

7. Builder 완료 Callback API 구현

Proxy Backend에 Builder 작업 결과를 받을 Callback API를 추가한다.

예:

POST /api/v1/builder/callback

성공 예시:

{
  "job_id": "job_123456",
  "file_id": "file_ce592ab151",
  "status": "COMPLETED",
  "wikis": [
    {
      "id": "wiki-data-cache",
      "title": "Data Cache",
      "slug": "data-cache",
      "summary": "Data Cache에 대한 설명",
      "content": "# Data Cache\n\n...",
      "metadata": {
        "keywords": [],
        "concepts": [],
        "entities": [],
        "relations": []
      }
    }
  ]
}

실패 예시:

{
  "job_id": "job_123456",
  "file_id": "file_ce592ab151",
  "status": "FAILED",
  "error": {
    "code": "LLM_API_ERROR",
    "message": "LLM API request failed"
  }
}

Callback 수신 시:

성공

Wiki 데이터 저장
→ file 상태 COMPLETED
→ builder job 상태 COMPLETED

실패

file 상태 FAILED
→ builder job 상태 FAILED
→ 오류 메시지 저장

Callback은 같은 요청이 여러 번 들어와도 데이터가 중복 생성되지 않도록 가능한 범위에서 idempotent하게 처리한다.

예:

job_id + file_id

기준으로 이미 완료 처리된 Callback인지 확인한다.

⸻

8. Frontend Polling 구조 유지

Frontend는 기존 API를 사용한다.

GET /api/v1/files/{file_id}

응답에 처리 상태를 포함한다.

처리 중:

{
  "file_id": "file_ce592ab151",
  "status": "PROCESSING",
  "stage": "ENRICHING"
}

완료:

{
  "file_id": "file_ce592ab151",
  "status": "COMPLETED",
  "wiki_ids": [
    "wiki-data-cache"
  ]
}

실패:

{
  "file_id": "file_ce592ab151",
  "status": "FAILED",
  "error_message": "LLM API request failed"
}

현재 Frontend polling 로직은 최대한 유지한다.

⸻

9. Timeout 설정

비동기 구조로 변경하더라도 Builder에 작업을 등록하는 HTTP 요청에는 명시적인 timeout을 설정한다.

예:

timeout = httpx.Timeout(
    connect=10.0,
    read=30.0,
    write=60.0,
    pool=10.0,
)

Builder의 ingest API는 작업만 등록하고 즉시 응답해야 하므로 read timeout을 수분 단위로 늘리지 않는다.

비동기 구조 변경 전 기존 코드를 임시로 유지해야 하는 경우에만 긴 timeout을 적용한다.

⸻

10. 예외 처리

다음 상황을 구분해서 처리한다.

Builder 연결 실패

BUILDER_UNAVAILABLE

파일 상태:

FAILED

Builder 작업 등록 실패

BUILDER_JOB_SUBMIT_FAILED

Builder 내부 처리 실패

Callback을 통해:

FAILED

Callback 처리 실패

오류 로그를 남기고, 동일 Callback 재호출 시 중복 생성 없이 다시 처리할 수 있도록 한다.

11. 로그 개선

다음 식별자가 모든 관련 로그에 포함되도록 한다.

file_id
job_id

예:

[builder-job] submitted file_id=file_ce592ab151 job_id=job_123456
[builder-job] processing file_id=file_ce592ab151 job_id=job_123456 stage=ENRICHING
[builder-job] completed file_id=file_ce592ab151 job_id=job_123456

실패:

[builder-job] failed file_id=file_ce592ab151 job_id=job_123456 error=...

12. 구현 시 중요 사항

* 기존 파일 업로드 기능을 깨뜨리지 않는다.
* 기존 Frontend polling API를 최대한 유지한다.
* 기존 Wiki 생성 로직 자체는 변경하지 않는다.
* 동기 실행 코드를 비동기 Job 실행으로 감싸는 방향으로 수정한다.
* 불필요한 대규모 리팩터링은 하지 않는다.
* Redis, Celery 등 새로운 인프라는 현재 프로젝트에 없다면 추가하지 않는다.
* 기존 상태 Enum, DB 모델, 서비스 계층을 우선 재사용한다.
* Builder와 Proxy Backend의 책임을 명확히 분리한다.
* 파일 시스템의 로컬 절대 경로를 서비스 간 API 계약으로 사용하지 않는다.
* 서비스 간 연결에는 file_id, job_id, API URL 등 논리적 식별자를 사용한다.

⸻

13. 완료 조건

다음 조건을 만족해야 한다.

1. 파일 업로드 후 Proxy Backend가 Builder 처리 완료까지 HTTP 연결을 유지하지 않는다.
2. Builder ingest API는 빠르게 202 Accepted를 반환한다.
3. Builder에서 Wiki 생성 작업이 별도로 계속 실행된다.
4. Frontend는 기존 polling 방식으로 처리 상태를 확인할 수 있다.
5. Wiki 생성 완료 후 Proxy Backend에 결과가 저장된다.
6. Wiki 생성 실패 시 FAILED 상태와 오류 정보를 확인할 수 있다.
7. 기존에 발생한 장시간 Builder 호출에 의한 httpx.ReadTimeout 문제가 제거된다.
8. 기존 Wiki 생성 결과와 기능은 변경되지 않는다.

구현 전 현재 backend-proxy와 wiki-builder의 관련 코드를 먼저 분석하고, 기존 구조와 네이밍을 최대한 유지한 최소 변경 방식으로 구현한다.
